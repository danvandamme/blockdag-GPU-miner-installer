## Bug 1 — GPU atomics silently dropped on NVIDIA (kernel produced no shares)

**File:** `dagtech_gpu.cl`

**Symptom:** GPU ran at full hashrate, reported H/s, but `output[1]` (found count) always stayed 0. No `** SHARE FOUND **` lines ever appeared. CPU shares accumulated normally.

**Root cause:** NVIDIA's OpenCL driver silently ignores `atomic_min` / `atomic_inc` on `__global volatile uint *` pointers. The `volatile` qualifier on the output buffer caused all atomic writes to be dropped at the driver level with no error.

**Fix — two changes to `dagtech_gpu.cl`:**

1. Add extension pragmas at the top of the file (before any function definitions):
```c
#pragma OPENCL EXTENSION cl_khr_global_int32_base_atomics     : enable
#pragma OPENCL EXTENSION cl_khr_global_int32_extended_atomics : enable
```

2. Remove `volatile` from the output parameter in the kernel signature:
```c
// BEFORE:
__kernel void dagtech_search(__global const uint *header80,
                             __global volatile uint *output,

// AFTER:
__kernel void dagtech_search(__global const uint *header80,
                             __global uint *output,
```

## Bug 2 — GPU pre-filter target ~1000x too lenient (shares found but all rejected by CPU re-verify)

**File:** `dagtech_miner.c`  
**Location:** `dagtech_gpu_thread()` function, inner batch loop (~line 644)

**Symptom:** After fixing Bug 1, the GPU started finding nonces (filling `output[0]`), but the CPU re-verify step almost always rejected them. At difficulty ~1.3, expected accepted GPU shares per 4 minutes ≈ 0.024 — effectively zero.

**Root cause:** The GPU pre-filter target was computed as:
```c
cl_uint target32 = (cl_uint)(0x00000000FFFFFFFFull / diff);
```
At difficulty 1.3 this gives `target32 ≈ 0xC5604189`. Since ~79% of all nonces produce a `hash[7]` below this threshold, `atomic_min` almost always recorded a nonce very close to `nonce_base`. That nonce had only a ~1/65536 chance of passing the strict CPU 64-bit check:
```
hash_top64 = ((uint64_t)BSWAP(ostate[7]) << 32) | BSWAP(ostate[6])
                 must be ≤ 0x0000FFFF00000000 / difficulty
```
The GPU target constant and the CPU threshold were using completely different scales.

**Fix — `dagtech_miner.c` ~line 650:**
```c
// BEFORE:
double diff = j.difficulty > 0.0 ? j.difficulty : 1.0;
cl_uint target32 = (cl_uint)(0x00000000FFFFFFFFull / diff);

// AFTER:
/* Compute 32-bit difficulty target matching the CPU's 64-bit check.
 * CPU checks: hash_top64 <= 0x0000FFFF00000000 / difficulty
 * hash_top64 upper 32 bits == hash[7] (BSWAP(ostate[7])), so the GPU
 * pre-filter target is the upper 32 bits of that threshold.
 * Using 0xFFFFFFFF/diff instead is ~1000x too lenient: atomic_min
 * always picks a nonce near nonce_base that rarely passes the CPU check. */
double diff = j.difficulty > 0.0 ? j.difficulty : 1.0;
double thresh_d = (double)0x0000FFFF00000000ULL / diff;
uint64_t thresh64 = (thresh_d >= 18446744073709551615.0) ? 0xFFFFFFFFFFFFFFFFULL : (uint64_t)thresh_d;
cl_uint target32 = (cl_uint)(thresh64 >> 32);
```

After this fix, at difficulty 1.3: `target32 ≈ 0x0000C5A8` — only ~0.08% of nonces pass the GPU filter, and all of those also satisfy the CPU's 64-bit check.

**After recompile and deploy:** GPU found 42 accepted shares in under 1 minute. The compiled `dagtech-gpu-miner.exe` in this directory already includes this fix.

## Bug 3 — Control server hung under scheduled task (dashboard showed "Miner offline")

**File:** `dagtech-control.ps1`

**Symptom:** Dashboard showed "⚠ Miner offline — control server unreachable" despite the control server process being alive. `netstat` showed 40+ CLOSE_WAIT connections accumulating on port 8883.

**Root cause:** The control server's `/status` handler used `Invoke-WebRequest -TimeoutSec 2` to fetch metrics from port 8882. In a Windows scheduled-task session (no interactive desktop, no proxy configured), `Invoke-WebRequest` can block indefinitely despite `TimeoutSec` — the .NET runtime behind it does not always honour the timeout in non-interactive sessions. Because `HttpListener` is single-threaded, one hung handler blocked all subsequent requests; http.sys kept accepting connections, which piled up as CLOSE_WAIT.

**Fix — replace `Invoke-WebRequest` with `HttpWebRequest` in `dagtech-control.ps1`:**

Find the block that fetches metrics (searches for `Invoke-WebRequest` targeting port 8882) and replace with:
```powershell
$metricsPort = if ($cfg["METRICS_PORT"]) { $cfg["METRICS_PORT"] } else { "8882" }
try {
    $wr = [System.Net.HttpWebRequest]::Create("http://127.0.0.1:$metricsPort/metrics")
    $wr.Timeout          = 2000
    $wr.ReadWriteTimeout = 2000
    $wr.Method           = "GET"
    $wr.Proxy            = $null
    $resp   = $wr.GetResponse()
    $reader = [System.IO.StreamReader]::new($resp.GetResponseStream(), [System.Text.Encoding]::UTF8)
    $mj     = $reader.ReadToEnd() | ConvertFrom-Json
    $reader.Close(); $resp.Close()
    foreach ($f in @("cpu_submitted","gpu_submitted","cpu_accepted","gpu_accepted","cpu_rejected","gpu_rejected","cpu_stale","gpu_stale")) {
        if ($null -ne $mj.$f) { $out[$f] = [long]$mj.$f }
    }
} catch {}
```

`[System.Net.HttpWebRequest]` with explicit `Timeout`, `ReadWriteTimeout`, and `Proxy = $null` always honours its timeout regardless of session context.

## Bug 4 — Per-source share fields missing from `/metrics` response

**File:** `dagtech-control.ps1`

**Symptom:** The `/metrics` JSON response had 21 keys and no `cpu_submitted`, `gpu_submitted`, `cpu_accepted`, `gpu_accepted`, `cpu_rejected`, `gpu_rejected`, `cpu_stale`, or `gpu_stale` fields. The dashboard "CPU / GPU Shares" card showed "--".

**Root cause:** These fields were not initialised in the `$out` hashtable that the control server builds for each `/metrics` response.

**Fix — add 8 fields to the `$out` hashtable** in `dagtech-control.ps1` (immediately after the `gpu_shares_found = 0` line):
```powershell
cpu_submitted    = 0
gpu_submitted    = 0
cpu_accepted     = 0
gpu_accepted     = 0
cpu_rejected     = 0
gpu_rejected     = 0
cpu_stale        = 0
gpu_stale        = 0
```

These are then populated by the `HttpWebRequest` fetch block (Bug 3 fix) which reads the values from the miner's own `/metrics` endpoint on port 8882.

## Dashboard change — CPU / GPU Shares card

**File:** `dashboard/index.html`

**Change:** Added a stat card showing accepted shares and submission counts for CPU and GPU separately. This is cosmetic — the miner works without it — but it makes it easy to see both sources at a glance.

Added HTML (after the CPU/GPU Hash Rate card):
```html
<div class="stat-card">
    <div class="stat-label">CPU / GPU Shares</div>
    <div style="display:flex;gap:18px;align-items:flex-start;margin-top:4px;">
        <div>
            <div style="font-size:10px;color:var(--text-dim);text-transform:uppercase;letter-spacing:0.5px;margin-bottom:3px;">CPU</div>
            <div class="stat-value blue" id="cpuSharesAccepted" style="font-size:18px;">--</div>
            <div id="cpuSharesSub" style="font-size:11px;color:var(--text-dim);margin-top:2px;">-- submitted</div>
        </div>
        <div>
            <div style="font-size:10px;color:var(--text-dim);text-transform:uppercase;letter-spacing:0.5px;margin-bottom:3px;">GPU</div>
            <div class="stat-value purple" id="gpuSharesAccepted" style="font-size:18px;">--</div>
            <div id="gpuSharesSub" style="font-size:11px;color:var(--text-dim);margin-top:2px;">-- submitted</div>
        </div>
    </div>
</div>
```

Added JS in `applyMetrics()`:
```javascript
const cpuAcc = data.cpu_accepted ?? null;
const gpuAcc = data.gpu_accepted ?? null;
const cpuAccEl = document.getElementById('cpuSharesAccepted');
const gpuAccEl = document.getElementById('gpuSharesAccepted');
const cpuSubEl = document.getElementById('cpuSharesSub');
const gpuSubEl = document.getElementById('gpuSharesSub');
if (cpuAccEl) cpuAccEl.textContent = cpuAcc !== null ? cpuAcc : '--';
if (gpuAccEl) gpuAccEl.textContent = gpuAcc !== null ? gpuAcc : '--';
if (cpuSubEl) {
    const sub = data.cpu_submitted ?? null;
    const rej = data.cpu_rejected || 0;
    const stl = data.cpu_stale || 0;
    cpuSubEl.textContent = sub !== null
        ? sub + ' sub' + (rej ? ' · ' + rej + ' rej' : '') + (stl ? ' · ' + stl + ' stale' : '')
        : '-- submitted';
}
if (gpuSubEl) {
    const sub = data.gpu_submitted ?? null;
    const rej = data.gpu_rejected || 0;
    const stl = data.gpu_stale || 0;
    gpuSubEl.textContent = sub !== null
        ? sub + ' sub' + (rej ? ' · ' + rej + ' rej' : '') + (stl ? ' · ' + stl + ' stale' : '')
        : '-- submitted';
}
```
